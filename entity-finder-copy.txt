INTEGRATION SECTION (How It Works)
--------------------------------------------------------------------------------
Section Label: Integration

Headline:
Submit. Poll.
Done.

Body: Built for async workflows. Submit a request, get a job ID back in 100 milliseconds. Poll when you're ready. No blocking. No waiting. Max 10 concurrent jobs. Results stay available for 24 hours.

API Flow Steps:

Step 01 — POST /api/v1/trace
Detail: Submit name + address. Receive a job_id in under 100ms.

Step 02 — Job runs asynchronously
Detail: AI agent pipeline executes up to 3 tiers. No blocking. No waiting on the client.

Step 03 — GET /api/v1/trace/{job_id}
Detail: Poll for status. When completed, retrieve fully structured JSON.

Step 04 — Structured result object
Detail: Every field is typed and nullable. Plug directly into your CRM, data warehouse, or workflow.

Example Request / Response:
POST /api/v1/trace
{
  "name": "Jane Doe",
  "address": "123 Main St, MI",
  "max_tier": 3
}

202 Accepted Response:
{
  "job_id": "a3f9c21b04d7",
  "status": "queued"
}

API REFERENCE SECTION (Collapsible documentation)
--------------------------------------------------------------------------------
Section Label: API Reference
Description: Full endpoint docs, data models, and job lifecycle

Authentication:
All /api/v1/* endpoints require an X-API-Key header.
GET /health does not require authentication.

Header: X-API-Key: <your-api-key>
401 Error: { "detail": "Invalid or missing API key" }

Endpoints:

GET /health
Description: Server status. No authentication required.
200 Response: { "status": "ok" }

---

POST /api/v1/trace
Description: Submit a new entity find job. Queued immediately, executed asynchronously. Deduplication: same name+address returns existing job_id.

Request body:
{
  "name": "Jane Doe",
  "address": "123 Main St, Detroit, MI 48201",
  "max_tier": 1
}

max_tier: 1 (default), 2, or 3. Omit for Tier 1 only.

202 Accepted Response:
{
  "job_id": "a3f9c21b04d7",
  "status": "queued"
}

---

GET /api/v1/trace/{job_id}
Description: Retrieve status and result of an entity find job. Poll until status is completed, failed, or cancelled.

200 Completed Response:
{
  "job_id": "a3f9c21b04d7",
  "status": "completed",
  "debtor": { "name": "...", "address": "...", "max_tier": 1 },
  "created_at": "2026-03-08T14:00:00Z",
  "completed_at": "2026-03-08T14:04:32Z",
  "total_steps": 87,
  "result": { ... },
  "error": null
}

200 Running/Queued Response: (result null)
404 Not Found: Job not found

---

DELETE /api/v1/trace/{job_id}
Description: Request cancellation. Acknowledged immediately (202); worker stops at next tier boundary. Poll GET to confirm status: cancelled and partial result.

202 Cancellation Requested:
{ "job_id": "...", "status": "running" }

409 Error: Job already completed/failed | 404 Error: Job not found

---

GET /api/v1/jobs
Description: List all jobs in memory. Returns full job details per job.
200 Response: Array of Job objects

Data Models:

DebtorInput (Request):
- name: string — Debtor's full name
- address: string — Known address (free-form)
- max_tier: integer — Max tier 1–3. Default 1.

TraceResponse (POST trace, DELETE trace):
- job_id: string
- status: queued | running | completed | failed | cancelled

Job (GET trace/{job_id}, GET jobs):
- job_id: string
- status: queued | running | completed | failed | cancelled
- debtor: DebtorInput
- created_at: datetime (UTC)
- completed_at: datetime | null
- total_steps: integer | null
- result: EntityFinderResult | null
- error: string | null

EntityFinderResult (result schema):
- full_legal_name: string | null
- aliases: string[]
- date_of_birth: string | null
- phone_numbers: PhoneNumber[]
- current_address: Address | null
- previous_addresses: AddressHistory[]
- email_addresses: string[]
- employment: EmploymentInfo | null
- spouse_partner: RelativeInfo | null
- co_habitants: string[]
- property_ownership: PropertyInfo | null
- homeowner_or_renter: owner | renter | unknown | null
- bankruptcy: BankruptcyInfo | null
- civil_judgments_liens: JudgmentLien[]
- voter_registration_address: string | null
- professional_licenses: License[]
- facebook_url: string | null
- estimated_income_range: string | null
- deceased: DeceasedInfo | null

Nested types: PhoneNumber (number, phone_type, carrier), Address (street, city, state, zip_code), AddressHistory (address, date_range), EmploymentInfo (employer_name, job_title, work_phone, employer_address, employer_main_line), RelativeInfo (name, phone, relationship), PropertyInfo (owner_of_record, estimated_value, estimated_equity, property_address), BankruptcyInfo (status, chapter, case_number, filing_date), JudgmentLien (case_type, description, amount, filing_date, court), License (license_type, status, issuing_state, license_number), DeceasedInfo (is_deceased, date_of_death, source).

Job Lifecycle:
POST /api/v1/trace → status: queued (returned immediately) → status: running → success: status: completed (result populated) OR error: status: failed (error populated)
DELETE during running → status: cancelled (partial result preserved)

Concurrency & Persistence:
- Max concurrency: 10 simultaneous trace jobs (asyncio.Semaphore)
- Persistence: Jobs in-memory only; history lost on restart
- Job TTL: Terminal jobs (completed/failed/cancelled) purged after 24 hours. Cleanup runs hourly.
- Result files: data/{job_id}.json written on completion; survive restarts

Known Issues & Optimizations:

Tiers Table:
| Tier | max_tier | Max steps | Finds |
| Tier 1 | 1 (default) | 30 | Name, phones, email, address, DOB, bankruptcy, deceased |
| Tier 2 | 2 | 30 | Employer, job title, work phone, aliases, previous addresses |
| Tier 3 | 3 | 40 | Spouse, co-habitants, property, judgments, licenses, voter reg, Facebook, income |

How to minimize token drain:
- Deduplication: Same name+address returns existing job_id; no duplicate pipeline.
- Cancel long-running jobs: DELETE /api/v1/trace/{job_id} returns 202; poll until status: cancelled and use partial result.
- Tier control: Use max_tier in request; default 1 runs Tier 1 only.
- Agent instructions: Tier 2/3 prompts tell the agent not to re-search already-confirmed fields.

Each tier runs only if the previous tier left target fields unfilled. Later tiers skip re-verifying already-found data.
